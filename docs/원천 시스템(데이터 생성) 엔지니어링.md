# 원천 시스템(데이터 생성) 엔지니어링

---

🔹 주요 엔지니어링 사항

1. 세 원천은 모두 외부 서드파티 시스템이므로, 스키마 변경·장애·rate limit을 통제할 수 없다는 전제로 설계 필요
2. `MK RSS`, `OpenDART`는 이벤트형 시계열, `KIS`는 측정형 시계열이므로 mart grain을 분리 필요
3. `MK RSS`는 최근 50개 항목만 유지되어 놓치면 복구가 어렵기 때문에, 짧은 주기 polling(10분)과 raw XML 보관 필요
4. `OpenDART`는 날짜 범위 백필, `KIS`는 종목×시간구간 백필이 가능하도록 처음부터 복구 단위 분리 필요
5. 원천 스키마를 그대로 downstream에 노출하지 말고, **bronze raw → silver standard** 변환 필요
6. 세 소스 모두 중복·누락 가능성이 있으므로, source별 중복 제거 키(dedup key), 재수집 기준점 필요
7. 모든 레이어에 `event_at`, `collected_at`, `processed_at`를 남겨야 지연, late arrival, 재처리 판단이 가능
8. 운영을 위해 “수집 성공 여부”가 아니라 row count, 중복률, 지연 시간, 스키마 변경, 품질 이상 알림을 갖춘 observability 필요
9. 오케스트레이션은 source별로 분리해야 하며, 특히 `KIS`는 인증 흐름 + 시장 시간 스케줄까지 고려한 설계 필요
10. 보안과 거버넌스는 secret을 코드 밖에서 관리하고, 품질 책임과 스키마 변경 대응을 내부 파이프라인 책임으로 설정

# 1. 원천 시스템 이해

## 1-1. 원천 식별

> 
> 
> - 원천의 종류가 다르면 수집 방식이 달라진다.
>     - ex. DB면 스냅샷 추출⋅CDC⋅읽기 부하가 핵심
>     - ex. API면 인증⋅페이지네이션⋅rate limit이 핵심
>     - ex. 파일이면 포맷⋅업로드 주기⋅수동 개입 여부가 핵심
> - 원천을 식별하면 파이프라인 설계를 시스템 특성에 맞게 할 수 있음

### 체크리스트

✅ 이 데이터의 원천 시스템은 무엇인가?

⇒ 셋 다 외부 시스템이므로 에러 수정⋅스키마 변경 방지⋅Rate Limit/인증정책 변경 불가

| **데이터 소스** | **원천 시스템** |
| --- | --- |
| MK RSS | 매일경제가 운영하는 RSS 제공 시스템 |
| OpenDART API | 금융감독원 전자공시시스템(DART)의 Open API 제공 시스템 |
| KIS OpenAPI | 한국투자증권이 운영하는 OpenAPI 제공 시스템 |

✅ 원천은 애플리케이션 DB / API / 파일 / 로그 / 메시지 큐 / 스트림 / IoT / 서드파티 데이터 중 무엇인가?

⇒ 셋 다 서드파티 데이터

| **데이터 소스** | 전달 방식 | **데이터 포맷** |
| --- | --- | --- |
| MK RSS | RSS 피드 엔드포인트
(HTTP API) | XML(RSS) |
| OpenDART API | HTTP API | JSON/XML |
| KIS OpenAPI | HTTP API | JSON |

✅ 데이터 접근 권한

| **데이터 소스** | 데이터 접근 권한 |
| --- | --- |
| MK RSS | 인증키 필요 없음 |
| OpenDART API | API 키 신청 후 사용 |
| KIS OpenAPI | 앱키, 앱시크릿 발급 후 접근 토큰을 받아서 사용 |

## 1-3. 데이터 보존성과 상태성

- 원천 시스템에서 데이터를 오래 보존하지 않으면 놓친 데이터를 다시 못 가져올 수 있음
- 백필 가능 여부를 미리 판단 가능
- 원천 특성에 맞는 저장 전략을 세울 수 있음(bronze 원본 보관 등)

### 체크리스트

✅ 데이터가 장기간 보존되는가, 아니면 빨리 사라지는가?

| **데이터 소스** | 데이터 보존성 | **비고** |
| --- | --- | --- |
| MK RSS | ❌ | 최근 50개만 유지되면서 오래된 항목을 밀어냄 |
| OpenDART API | ⭕️ | 조회 시작일, 조회 마지막일을 파라미터로 받고, 과거 데이터도 조회 가능 |
| KIS OpenAPI | ⭕️ | 기간별 데이터 조회 가능, 다만 보존 정책에 관련된 내용이 없어서 보수적으로 봐야함 |

✅ 데이터가 삭제되면 복구 가능한가?

| **데이터 소스** | 데이터 복구 | **비고** |
| --- | --- | --- |
| MK RSS | ❌ | 삭제된 아이템 복구나 재조회 방식이 없음 |
| OpenDART API | ⭕️ |  |
| KIS OpenAPI | ⭕️(부분 가능) | 호출 건수 제한이 있어 대량 복구는 나눠서 해야함 |

# 3. 시간과 수집 전략을 점검하는 체크리스트

## 3-1. 발생 시간과 처리 시간

- “원천 데이터에서 언제 발생했는가”는 “언제 도착했는가”와 다름
- 이 구분이 없으면 지연, 누락, 재수집 판단이 불가

| 체크리스트 | MK RSS | OpenDART API | KIS OpenAPI |
| --- | --- | --- | --- |
| event time이 있는가? | ⭕️ (`pubDate`) | ⭕️ (`rcept_dt` 등 공시 시점) | ⭕️ (가격 기준 시각/일자) |
| ingestion time을 기록하는가? | 내부에서 반드시 기록 | 내부에서 반드시 기록 | 내부에서 반드시 기록 |
| process time을 기록하는가? | 내부에서 기록 권장 | 내부에서 기록 권장 | 내부에서 기록 권장 |
| late arrival 판별 기준 시간이 있는가? | ⭕️, `pubDate vs collected_at` | ⭕️, `공시시각 vs collected_at` | 낮음, `관측시각 vs collected_at` 정도 |

## 3-2. 수집 주기와 동기화 전략

| 체크리스트 | MK RSS | OpenDART API | KIS OpenAPI |
| --- | --- | --- | --- |
| 얼마나 자주 가져와야 하는가? | 10분 주기
(50건이 약 55시간에 걸쳐 수집됐지만, 특정 시간 대(ex. 금요일 오후 6시)에 몰려 있는 경우도 있으므로) | 하루 1회 수집, 페이지네이션되어 있으므로 여러 번 호출 필요 | 요구화면의 그래프를 그리기 위해 분 단위 수집 |
| full extract vs incremental | incremental + dedup | 날짜 기준 incremental 적합 | 시점/일자 기준 incremental 적합 |
| snapshot vs CDC | snapshot-like feed | snapshot 조회 API | snapshot 조회 API |
