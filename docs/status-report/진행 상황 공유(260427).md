### 설명/시연 계획

#### DAG 구조 설명

- Airflow UI Graph View : `raw -> bronze -> silver -> mart` task 의존성
- `KIS 현재가`: 1분 주기, `catchup=False`, 장중 여부 `short_circuit`
- `MK RSS`: 10분 polling, `catchup=False`, RSS 휘발성
- `KIS 주가 history`: 일봉 backfill용 DAG, `catchup=True`, `{{ ds_nodash }}` 기준 날짜 조회 설명
- 공통: `max_active_runs=1`, `retries=2`, `retry_delay=1분`, task `execution_timeout` 설명

#### 정상 실행 1번

- 예시 : `MK RSS` DAG 수동 실행
  - Airflow UI : DAG run 성공 상태, Graph View의 task 성공 흐름
- raw/bronze/silver/mart 산출물이 생성된 경로 또는 mart 조회 결과 제시

#### 실패 후 retry/복구 1번

- Airflow가 `retries=2`, `retry_delay=1분` 기준으로 실패 task만 재시도

#### 같은 날짜 재실행(backfill) 시 결과 보장 방식

- `KIS 주가 history`는 `{{ ds_nodash }}`를 조회 날짜로 사용하므로 같은 날짜 run은 항상 같은 날짜 데이터를 조회
- backfill 날짜 범위를 지정하면 schedule 기준으로 날짜별 DAG run이 생성
- `max_active_runs=1`로 날짜별 run을 순차 실행해 DuckDB 적재 충돌 방지
- mart 적재는 비즈니스 키 기준 중복 방지 조건을 사용하므로 같은 날짜를 다시 실행해도 중복 적재를 막는 구조

# 3. Backlog

## 3.1 데이터 모델링

- Silver 표준 구조 정의
- 조회 요구사항 기준 Mart DDL 설계
- 원천별 데이터 특성을 반영한 grain·key·적재 기준 확정

## 3.2 데이터 저장 주요 엔지니어링

- 현재 로컬 파일시스템 기반으로 Bronze·Silver 저장 중
- 운영 안정성, 재처리, 확장성을 고려한 S3 도입 검토
- S3 도입 필요성과 기대효과를 비용 관점까지 포함해 문서화

## 3.3 데이터 변환 주요 엔지니어링

- 비즈니스 목표: 주가와 주요 경제 이벤트를 함께 제공해 사용자의 경제 흐름 이해 지원
- 과제: 어떤 이벤트를 “주요”로 볼 것인지에 대한 기준 정립 필요
- 해셕 가능한 형태의 데이터 중 어떤 것이 주요한 데이터인지 선별 필요
