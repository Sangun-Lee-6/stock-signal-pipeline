# 진행 상황 공유 (260507)

## 1. 프로젝트 개요 및 목적

- 이 프로젝트는 주가 데이터와 경제 뉴스 이벤트를 함께 수집하고, 정제한 뒤 웹 화면에서 확인할 수 있게 만든 데이터 엔지니어링 파이프라인이다.
- 외부 데이터를 안정적으로 수집하고, 재처리 가능한 형태로 저장한 뒤, 서비스 조회까지 연결하는 것을 목표로 했다.
- 데이터 엔지니어링 수명주기에 맞춰 원천 수집부터 서빙까지의 흐름을 설계했다.
  - 기술을 많이 붙이는 것보다, 수집-저장-변환-적재-서빙 흐름에서 각 단계의 책임을 분리하는 데 집중했다.
  - `raw -> bronze -> silver -> mart -> serving` 흐름으로 운영 가능한 데이터 파이프라인을 구성했다.
- 최근에는 장애 감지, 품질 체크, 실패 알림, DuckDB 동시 쓰기 제어처럼 운영 안정성에 가까운 부분을 보강했다.

## 2. 전체 흐름

![pipeline-architecture](../image/pipeline-architecture.png)

```text
KIS 현재가 / KIS 일봉 / MK RSS
  -> raw
  -> bronze
  -> silver
  -> silver_created_manifest
  -> load_silver_to_mart
  -> DuckDB mart / serving view
  -> FastAPI web UI
```

- `raw`는 원천 응답 보존, `bronze`는 저장 형식 표준화, `silver`는 분석 가능한 정제 데이터, `mart`는 조회 목적 테이블로 역할을 나눴다.
- `silver_created_manifest`를 중간 계약으로 두어 수집 DAG와 mart 적재 DAG가 직접 강하게 묶이지 않도록 했다.
- 최종적으로는 DuckDB serving view를 FastAPI 웹 앱이 read-only로 조회하는 구조다.

## 3. 이번 진행 상황

### 3.1 운영 안정성 보강

- raw, bronze, silver 단계별 품질 체크 결과를 JSON으로 남기도록 구성했다.
- 품질 체크 실패 시 조용히 넘어가지 않고 task를 실패시켜 Airflow retry와 실패 추적이 가능하게 했다.
- Google Chat 실패 알림 callback을 추가해 DAG/task 실패를 외부로 알릴 수 있게 했다.
- `check_pipeline_health` DAG로 최근 품질 체크 실패와 pending silver backlog를 점검할 수 있게 했다.
- 이 구조를 통해 성공 여부뿐 아니라 실패 단계와 영향 범위를 task, 파일 단위로 확인할 수 있다.

## 4. 시연 흐름

1. Airflow UI에서 주요 DAG 구조 확인
   - `collect_kis_stock_price_raw`
   - `collect_mk_rss_raw`
   - `load_silver_to_mart`
   - `check_pipeline_health`
   - 각 DAG의 책임과 분리 기준 확인

2. 데이터 산출물 확인
   - raw/bronze/silver 파일
   - `silver_created_manifest`
   - 품질 체크 결과 JSON
   - DuckDB mart 적재 결과
   - 파일 단위 lineage와 재실행 시 중복 적재 방지 흐름 확인

3. 웹 화면 확인
   - 주가 타임라인
   - 관련 뉴스 이벤트
   - 영향 범위별 이벤트 분류
   - 파이프라인 결과의 서비스 화면 반영 확인

## 5. 주요 설계 판단

- source별 데이터 특성이 달라 DAG schedule과 catchup 전략을 분리했다.
  - KIS 현재가: 장중 1분 snapshot
  - MK RSS: 10분 polling
  - KIS 일봉: 날짜 단위 backfill
- 수집 DAG와 mart 적재 DAG를 분리해 DuckDB write 충돌 가능성을 줄였다.
  - [DuckDB 동시 쓰기 문제 해결 과정](<../data-engineering-lifecycle/DuckDB 동시 쓰기 문제 해결 과정.md>)
- 파일 단위 manifest와 loaded marker를 사용해 재시도와 재실행에 강한 구조로 만들었다.
- DuckDB write 경로에는 Airflow pool, OS file lock, transaction을 함께 사용해 scheduler 수준과 process 수준에서 동시 쓰기를 제어했다.
- 뉴스 이벤트 분류는 외부 AI 호출 없이 결정적 규칙 기반으로 처리해 비용, 재현성, 검증 가능성을 우선했다.
  - [데이터 변환 엔지니어링(해석 가능한 형태로 변환하기)](<../data-engineering-lifecycle/데이터 변환 엔지니어링(해석 가능한 형태로 변환하기).md>)
- 전체 설계 기준은 source별 데이터 특성, DuckDB의 embedded DB 제약, 재처리 가능성이다.

## 6. 현재 한계

- OpenDART 수집 DAG는 구조만 있고 현재 비활성화 상태다.
- 로컬 S3-style 디렉터리를 사용하고 있어 실제 S3 배포 검증은 아직 필요하다.
- 현재 수집 대상 종목 확장은 설정화가 더 필요하다.
- 뉴스 이벤트 분류 품질을 평가할 테스트 데이터셋과 지표가 아직 부족하다.
- 별도 metric 저장소나 SLA 대시보드는 아직 없다.
- 다만 현재 구조는 source 추가, mart 확장, 실제 object storage 전환을 염두에 두고 레이어를 분리해둔 상태다.

## 7. 회고

- 가장 많이 배운 부분은 Airflow가 단순 스케줄러가 아니라 실패, 재시도, 의존성, 재처리 범위를 관리하는 오케스트레이션 도구라는 점이다.
- DuckDB 동시 쓰기 문제를 직접 재현하고, pool과 file lock, transaction을 조합해 해결하면서 embedded DB를 파이프라인에서 사용할 때의 제약을 이해했다.
- 다음에 다시 한다면 초기에 source별 grain, key, 적재 기준을 더 명확히 정리하고 데이터 품질 검증 기준도 함께 설계할 것 같다.
