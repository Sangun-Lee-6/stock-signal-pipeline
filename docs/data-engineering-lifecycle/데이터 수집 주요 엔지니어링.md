# 데이터 수집 주요 엔지니어링

- 목적 : 데이터를 가져오고 끝내는게 아니라 비즈니스 사용 사례에 맞는 방식으로⋅안정적으로⋅운영 가능하게 가져오는 설계
- 현재 source별 특성을 반영해서 수집 중(raw 원본 보존 및 재처리 경계 확보), 안정적 운영은 아직 미흡함

# 0. 데이터 수집 현황 요약

- `KIS(한국투자증권)`
    - 주가 타임라인 구축용 1분 단위 스냅샷 수집
    - 현재 1종목 하드코딩 기반의 확장성 한계
- `MK(매일경제) RSS`
    - 뉴스 이벤트 후보 확보용 10분 polling 수집
    - 최근 50개 기사만 조회되는 롤링 윈도우
    - RSS XML snapshot 전체 raw 저장 방식
    - source 휘발성으로 인한 완전 복구 한계
- `OpenDART`
    - 기업 이벤트의 날짜 기준(일 1회) 수집
    - manifest(수집 계획) 선생성 후 page 단위 확장 방식
        - (⚠️ XCom에 큰 데이터를 넘기지 않도록 검토 필요)
    - 빈 날짜(status=013) 정상 처리 구조 : 데이터 누락과 부재를 구분

---

1. 수집 목적과 범위
    1. 사용 사례 정의
        - `KIS` : 종목 가격 타임라인
        - `MK RSS`, `OpenDART` : 뉴스/공시 이벤트 타임라인
2. 수집 방식 결정
    1. 배치 vs 스트리밍
        - KIS 1분(그래프를 위해), MK RSS는 10분(생성 패턴 조사), OpenDART는 하루 1회 수집
        - 마이크로 배치로 충분
    2. 푸시 vs 풀 vs 폴링
        - 폴링 기반 풀, Airflow가 주기적으로 데이터 수집
        - rate limit
            - `KIS` : 1초 18건까지 호출 가능 → 1분 1건 호출
            - `MK RSS` : rate limit 정보 없음 → 10분 1회 호출
            - `OpenDART` : 하루 20,000건/분당 1,000회 → 하루 1회 호출
        - 증분 조회
            - `OpenDART`는 `bgn_de/end_de/page_no`가 증분 기준
            - `KIS`는 `stock_code x collected_at` 시점 스냅샷 기준
            - `MK RSS`: 증분 조회보다 피드 스냅샷 폴링 구조 → dedup은 처리하지 않음
    3. 동기 vs 비동기
        - Dag 안에서 수집 흐름이 동기식
        - ⚠️ `raw -> bronze -> silver -> mart`가 한 DAG에 묶여 있어 하나의 실패가 전체 실패로 이어짐(DuckDB lock)
3. 수집 빈도와 데이터 흐름 특성
    1. 유한 데이터 vs 무한 데이터
        - 시간이 계속 흐르며 생성되는 무한 데이터
        - 수집 시점을 경계로 무한 데이터를 자름(`KIS`는 `1분 run`, `MK RSS`는 `10분 snapshot`, `OpenDART`는 `일자 + page` )
    2. 수집 빈도
        - 수집 빈도는 비즈니스 사용 사례와 맞음
4. 적재 패턴과 추출 전략
    1. 전체 스냅숏 vs 차등 추출
        - `MK RSS`와 `KIS`는 수집 시점 snapshot
        - `OpenDART`는 날짜 범위와 page 단위 증분 조회
5. 신뢰성, 내구성, 복구
    1. 신뢰성
        - 재시도, timeout 처리는 완료
        - ⚠️ 소스 장애 시 스킵/알림 정책 없음
    2. 내구성
        - 재처리
            - `OpenDART`, `KIS`는 재조회 여지가 있어 재처리 가능
            - `MK RSS`는 원천이 휘발성이어서 놓친 구간의 완전 복구가 어려움
        - 멱등성
            - silver/mart 는 멱등성을 고려(가격 시점, 기사 ID, 공시 ID 기준 중복 방지)
            - ⚠️ 전체 DAG 수준에서는 mart lock 실패로 완전한 운영 멱등성 보완 필요
    3. 복구 전략
        - ⚠️ 명확한 백필 전략이 필요
        - ⚠️ 부분 실패 시 자동 재처리 절차가 없음
6. 운영과 관찰 가능성
    1. 모니터링
        - ⚠️ 수집 성공/실패를 모니터링 필요
    2. 로그와 추적성
        - `collection_id`가 배치/run 추적 ID 역할을 하고 소스별 비즈니스 키도 따로 존재함

# 1. 수집 목적과 범위

## 1-1. 사용 사례 정의

- 사용 사례가 불명확하면 수집 빈도, 저장소, 변환 방식, 실시간 여부를 전부 잘못 결정할 수 있다.

✅ 왜 이 데이터를 수집하는지 **비즈니스 목적**으로 설명할 수 있는가?

- `KIS` : 종목 가격 타임라인을 만들기 위해 수집
- `MK RSS, OpenDART` : 가격 변화와 함께 볼 뉴스/공시 이벤트 타임라인을 만들기 위해 수집
- 보완 필요: 현재 `KIS`는 `LX세미콘 1종목`만 수집 → MVP 구현 후 종목 확장 필요

---

# 2. 수집 방식 결정

## 2-1. 배치 vs 스트리밍

- 실시간성은 운영 복잡성이 크다.

✅ 실시간이 정말 필요한가, 아니면 **마이크로 배치**로 충분한가?

- 마이크로 배치로 충분함
- `KIS`는 1분, `MK RSS`는 10분, `OpenDART`는 하루 1회로 수집하므로 실시간 프레임워크가 아닌 Dag 호출로 가능

---

## 2-2. 푸시 vs 풀 vs 폴링

✅ 수집 방식이 **푸시(push)** 인가, **풀(pull)** 인가, **폴링(polling)** 인가?

- 현재 폴링 기반 pull, Airflow가 주기적으로 외부 API/RSS를 호출해서 데이터 수집

✅ 폴링 주기가 너무 짧아 원천 시스템을 괴롭히지 않는가?

- `KIS` : 1초 18건까지 호출 가능 → 1분 1건 호출
- `MK RSS` : rate limit 정보 없음 → 10분 1회 호출
- `OpenDART` : 하루 20,000건/분당 1,000회 → 하루 1회 호출

✅ 풀 기반이면 증분 조회 기준(예: timestamp, id, offset)이 명확한가?

- `OpenDART`는 `bgn_de/end_de/page_no`가 증분 기준
- `KIS`는 `stock_code x collected_at` 시점 스냅샷 기준
- `MK RSS`는 증분 조회보다 피드 스냅샷 폴링 구조
    - 최신 50개만 유지되는 휘발성 source라 raw bronze에서는 dedup보다 snapshot 보존을 우선
    - 현재 규모에서는 저장 비용보다 기사 누락 리스크가 더 큼
    - dedup은 raw가 아니라 silver 또는 후속 레이어에서 기사 단위로 처리하는 방향

---

## 2-3. 동기 vs 비동기

✅ 수집 파이프라인이 **동기식**인가, **비동기식**인가?

- Dag 안에서 Task의 흐름은 동기식 직렬 실행

❌ 한 단계 실패가 전체 파이프라인 재실행으로 이어지는가?

- 현재는 source별로 `raw -> bronze -> silver -> mart`가 한 DAG에 묶여 있어 한 단계의 실패가 전체 실행 실패로 이어짐
- raw는 저장됐지만, mart 적재 단계에서 DuckDB lock으로 실패한 실행이 있음 ⇒ raw 수집 DAG와 silver/mart 적재 DAG 분리 필요

---

# 3. 수집 빈도와 데이터 흐름 특성

## 3-1. 유한 데이터 vs 무한 데이터

✅ 이 데이터는 본질적으로 **무한(unbounded)** 인가, 아니면 **유한(bounded)** 인가?

- 세 소스 모두 시간이 계속 흐르며 생성되는 `unbounded` 데이터

✅ 무한 데이터를 어느 경계에서 배치로 자를지 정의했는가?

- `KIS`는 `1분 run`, `MK RSS`는 `10분 snapshot`, `OpenDART`는 `일자 + page` 기준으로 자름

---

## 3-2. 수집 빈도

✅ 수집 빈도가 사용 사례와 맞는가?

- `KIS 1분`, `MK RSS 10분`, `OpenDART 하루 1회`는 source 성격과 일치

# 4. 적재 패턴과 추출 전략

## 4-1. 전체 스냅숏 vs 차등 추출

✅ 전체 스냅숏(full snapshot) 방식인가?

- `MK RSS`와 `KIS`는 수집 시점 snapshot

✅ 증분/차등 추출(incremental/differential update) 방식인가?

- `OpenDART`는 날짜 범위와 page 단위 증분 조회

---

# 5. 신뢰성, 내구성, 복구

## 5-1. 신뢰성

❌ 재시도, timeout, circuit breaker 같은 기본 보호 장치가 있는가?

- `retries`, `retry_delay`, `request timeout`, 일부 source 재시도는 있음
- 보완 필요: circuit breaker, source 장애 시 skip/alert 정책은 아직 없음

---

## 5-2. 내구성

✅ 수집 실패 시 데이터가 유실되는가, 아니면 재처리 가능한가?

- `OpenDART`, `KIS`는 재조회 여지가 있어 재처리 가능
- `MK RSS`는 원천이 휘발성이어서 놓친 구간의 완전 복구가 어려움

❌ 멱등성(idempotency) 처리가 가능한가?

- silver/mart 는 멱등성을 고려(가격 시점, 기사 ID, 공시 ID 기준 중복 방지)
- 전체 DAG 수준에서는 mart lock 실패로 완전한 운영 멱등성 보완 필요

---

## 5-3. 복구 전략

❌ 장애 발생 시 백필(backfill) 전략이 있는가?

- 명확한 백필 전략이 필요

❌ 부분 실패 시 전체 재실행이 아니라 구간 재실행이 가능한가?

- `OpenDART`는 날짜/page 단위, `KIS`는 종목/시점 단위로 재실행 경계를 설정할 수 있지만
- 자동화된 재처리 절차는 아직 없음

---

# 6. 운영과 관찰 가능성

## 6-1. 모니터링

❌ 수집 성공/실패를 모니터링하는가?

- Airflow task log와 실행 상태로 수동 확인은 가능하지만 운영용 메트릭, 소스 장애 알림 등은 없음

---

## 6-2. 로그와 추적성

✅ 레코드/배치/파일 단위 추적 ID가 있는가?

- `collection_id`가 배치/run 추적 ID 역할을 하고 소스별 비즈니스 키도 따로 존재함
- `KIS`는 `stock_code + collected_at`, `MK RSS`는 `article_id`, `OpenDART`는 `disclosure_id` 기준으로 추적 가능
