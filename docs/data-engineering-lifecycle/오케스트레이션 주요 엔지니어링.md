# 오케스트레이션 주요 엔지니어링

> 오케스트레이션은 단순히 “정해진 시간에 실행하는 스케줄링”이 아니라, 데이터 작업들의 실행 순서·의존성·상태·실패·재실행을 관리하는 시스템이다.

# 1. 스케줄링

## 핵심 문제

`stock-signal-pipeline`은 KIS 현재가, MK RSS, OpenDART처럼 성격이 다른 source를 수집한다.
각 source는 데이터 생성 주기와 복구 가능성이 다르기 때문에 하나의 공통 주기로 실행하면 안 된다.

## Airflow가 해결하는 방식

DAG별 `schedule`, `catchup`을 통해 source별 실행 주기와 재처리 전략을 분리한다.

## 프로젝트 적용

### `KIS 현재가`

- [1분 단위 주가 스냅샷 수집해야 함] => dag 스케줄링(1분마다 dag run 실행 생성)으로 해결

```Python
@dag(
    dag_id="collect_kis_stock_price_raw",
    schedule="* 9-15 * * 1-5",  # 평일 09:00~15:59에 1분마다 실행 생성
    ...
)
```

- [과거 시점 복구가 어려우므로 과거 스냅샷을 재실행하지 않아야 한다.] => `catchup=False`

- [수집할 필요가 없는 시간에는 작업을 실행하지 않아야 한다.] => `@task.short_circuit`
  - short_circuit task는 False를 반환하면 downstream task를 모두 실행하지 않음
  - 장 운영 시간에만 작업이 실행되도록

### `MK RSS`

- [10분 단위 polling] => dag 스케줄링(10분마다 dag run 실행 생성)으로 해결

- [최신 기사만 유지되는 휘발성 소스이므로 과거 시점 복구가 어렵다.] => `catchup=False`
  - 불필요한 작업 실행 방지

# 2. 의존성 관리

## 핵심 문제

데이터 파이프라인은 수집, 정제, 변환, 적재가 순서대로 실행되어야 한다.
앞 단계 산출물이 준비되지 않았는데 다음 단계가 실행되면 잘못된 데이터가 만들어지거나, 실패 원인을 추적하기 어려워진다.

## Airflow가 해결하는 방식

Airflow는 task를 DAG 안의 node로 정의하고, `>>` 연산자로 task 실행 순서를 명시한다.
이렇게 하면 upstream task가 성공해야 downstream task가 실행된다.

## 프로젝트 적용

### 공통

- [원본 데이터를 먼저 보존해야 함] => raw 저장 task를 가장 앞에 둔다.

- [raw 데이터를 표준화한 뒤 분석 가능한 형태로 정제해야 함] => bronze/silver task를 raw 이후에 실행한다.

- [정제된 데이터를 바탕으로 화면이나 분석에 사용할 mart를 만들어야 함] => mart task를 마지막에 실행한다.

- [각 단계가 순서대로 실행되어야 함] => `>>` task dependency로 실행 순서를 고정한다.

```Python
raw >> bronze >> silver >> mart
```

# 3. 재시도

## 핵심 문제

외부 API/RSS 수집은 네트워크 지연, 일시적인 원천 서버 오류, 응답 timeout 때문에 실패할 수 있다.
이런 일시적 실패마다 전체 파이프라인을 사람이 직접 다시 실행하면 운영 부담이 커지고, 수집 구간이 비어 있을 수 있다.

## Airflow가 해결하는 방식

Airflow는 task 실패 시 `retries`, `retry_delay` 설정에 따라 같은 task를 자동으로 다시 실행한다.
재시도는 task 단위로 동작하므로, 실패한 작업만 다시 시도하고 upstream/downstream 의존성 상태는 Airflow가 관리한다.

## 프로젝트 적용

### 공통

- [외부 API/RSS 요청은 일시적으로 실패할 수 있음] => `default_args`으로 재시도 횟수를 설정

- [실패 직후 바로 재요청하면 원천 서버 상태가 회복되지 않았을 수 있음] => `retry_delay`로 재시도 전 대기 시간을 설정

- [외부 요청 1건이 오래 멈추는 문제] => plugin request timeout=30으로 방지

### 데이터 소스별 request timeout 설정

- [외부 요청 1건이 오래 멈추는 문제] => plugin의 `request.urlopen(..., timeout=30)`으로 방지

- [Airflow task 전체가 오래 붙잡히는 문제] => DAG task에 `execution_timeout`을 설정

- [외부 데이터를 수집하는 task] => request timeout과 execution timeout을 모두 둔다.

- [bronze/silver 변환 task] => 외부 요청은 없지만 파일 I/O나 변환 지연으로 오래 붙잡히지 않도록 execution timeout을 둘 수 있다.

- [mart 적재 task] => DuckDB write lock이나 적재 지연으로 task가 오래 대기할 수 있으므로 execution timeout을 둔다.

- [KIS 현재가는 1분 단위 DAG라 task가 오래 걸리면 다음 실행에 영향을 줄 수 있음] => 각 수집/적재 task에 `execution_timeout=timedelta(seconds=50)` 설정

- [MK RSS는 10분 단위 DAG라 KIS보다 여유가 필요함] => 각 수집/적재 task에 `execution_timeout=timedelta(minutes=3)` 설정

# 4. 백필

## 핵심 문제

일봉 데이터는 날짜 단위로 다시 수집할 수 있어야 한다.
특정 날짜 수집이 실패했거나, 변환 로직이 바뀌었거나, 과거 기간 데이터를 처음부터 채워야 할 때 사람이 날짜별로 직접 실행하면 누락과 중복이 생기기 쉽다.

## Airflow가 해결하는 방식

Airflow는 `catchup=True`일 때 `start_date`와 `schedule` 기준으로 과거 DAG run을 생성한다.
각 DAG run은 자신의 `logical_date`를 가지므로, task가 이 날짜를 조회 파라미터로 사용하면 날짜별 재처리가 가능하다.

## 프로젝트 적용

### `KIS 주가 history`

- [과거 주가 데이터를 날짜별로 다시 수집해야 함] => `catchup=True`로 과거 미실행 run을 생성한다.

- [각 backfill run이 자기 날짜의 데이터를 조회해야 함] => `{{ ds_nodash }}`를 KIS 일봉 조회 시작일/종료일로 넘긴다.

```Python
@dag(
    dag_id="collect_kis_stock_price_daily_history_raw",
    schedule="0 19 * * 1-5",
    catchup=True,
    ...
)
```

```Python
collect_raw_task = task(collect_daily_history_raw_task)(
    target_start_date="{{ ds_nodash }}",
    target_end_date="{{ ds_nodash }}",
)
```

- [2026-03-25 run] => `20260325` 일봉 데이터를 조회한다.

- [2026-03-26 run] => `20260326` 일봉 데이터를 조회한다.

### 백필 대상으로 보지 않는 데이터

- [`KIS 현재가`] => 현재 시점 스냅샷 수집이므로 `catchup=True`를 켜도 과거 1분 현재가를 복구하지 못한다.

- [`MK RSS`] => RSS feed가 최신 기사만 유지되는 휘발성 source라 과거 feed snapshot을 복구하지 못한다.

- [따라서 현재가/RSS의 수집 공백] => backfill이 아니라 `# 8. 수집 공백 감지와 보정`에서 별도 gap 처리 문제로 다룬다.

# 5. 동시성 제어

## 핵심 문제

짧은 주기로 실행되는 DAG는 이전 실행이 끝나기 전에 다음 실행이 시작될 수 있다.
같은 source의 run이 겹치면 동일 데이터를 중복 수집하거나, 같은 파일/테이블에 동시에 쓰면서 충돌이 날 수 있다.

특히 이 프로젝트는 DuckDB mart 적재가 포함되어 있어 여러 task가 동시에 write하면 lock 문제가 생길 수 있다.

## Airflow가 해결하는 방식

Airflow는 DAG 단위로 `max_active_runs`를 설정해 같은 DAG의 동시 실행 개수를 제한한다.
`max_active_runs=1`이면 이전 DAG run이 끝나기 전에는 다음 run이 동시에 실행되지 않는다.

task 단위에서는 `execution_timeout`으로 오래 붙잡힌 task를 실패 처리해서 다음 실행이 계속 밀리는 상황을 줄인다.

## 프로젝트 적용

### 공통

- [같은 source의 DAG run이 겹치면 중복 수집/중복 적재가 발생할 수 있음] => 모든 주요 수집 DAG에 `max_active_runs=1`을 설정한다.

- [raw, bronze, silver, mart가 같은 DAG 안에서 순서대로 실행되어야 함] => 한 DAG run이 끝난 뒤 다음 run이 실행되도록 제한한다.

- [DuckDB mart 적재가 동시에 발생하면 write lock 위험이 있음] => 같은 DAG 내부에서는 `max_active_runs=1`로 동일 source의 동시 적재를 막는다.

```Python
@dag(
    max_active_runs=1,
    ...
)
```

## 현재 한계

- [서로 다른 DAG가 동시에 mart 적재를 실행하는 문제] => `max_active_runs=1`은 같은 DAG 안에서만 동작하므로 완전히 막지 못한다.

- [source 전체의 API 호출량을 통합 제어하는 문제] => 현재 Airflow pool 기반 rate limit은 아직 없다.

- [다음 개선] => DuckDB 적재 task 전용 pool 또는 mart 적재 DAG 분리를 검토한다.

# 6. 동적 태스크

## 핵심 문제

OpenDART 공시 목록은 날짜마다 조회되는 데이터 양이 다르다.
어떤 날은 1 page만 있을 수 있고, 어떤 날은 여러 page가 필요할 수 있다.

수집해야 할 page 수를 미리 고정하면 page가 부족한 날에는 불필요한 task가 생기고, page가 많은 날에는 일부 데이터를 놓칠 수 있다.

## Airflow가 해결하는 방식

Airflow의 dynamic task mapping은 upstream task 결과를 기준으로 실행 시점에 task 개수를 동적으로 만든다.
즉, 먼저 manifest task에서 전체 page 수를 확인하고, 그 page 수만큼 page 수집 task를 생성할 수 있다.

## 프로젝트 적용

### `OpenDART`

- [날짜마다 공시 page 수가 달라짐] => 먼저 page manifest를 수집한다.

- [manifest에서 `total_page`를 확인해야 함] => `build_page_requests`에서 page별 요청 목록을 만든다.

- [page 수만큼 수집 task를 실행해야 함] => `collect_raw_page.expand(...)`로 page별 task를 동적으로 생성한다.

- [page별 raw 저장/변환/적재도 같은 개수로 이어져야 함] => bronze/silver/mart 단계도 `.expand(...)`로 연결한다.

## 현재 상태

- OpenDART page 단위 dynamic task mapping 구조는 DAG에 구현되어 있다.
  - 현재 `collect_page_manifest` task가 `AirflowSkipException`을 발생시키므로 OpenDART 수집은 비활성화되어 있다.

## 현재 한계

- [page별 task가 많아질 수 있음] => source API rate limit을 고려한 pool/concurrency 설정이 필요

- [page별 mart 적재가 동시에 발생할 수 있음] => DuckDB write lock을 피하려면 mart 적재 단계는 별도 처리

- [XCom에 큰 데이터를 넘길 위험] => page request처럼 작은 메타데이터만 넘기고, 큰 raw 데이터는 파일 경로로 전달하는 구조를 검토

# 7. 관측 가능성

## 핵심 문제

데이터 파이프라인은 실패했을 때 "어디서 실패했는지"와 "어떤 데이터 구간이 영향을 받았는지"를 빠르게 확인할 수 있어야 한다.
수집, 저장, 변환, 적재가 한 번에 묶여 있으면 실패 원인을 로그 없이 추측하게 되고, 재실행 범위도 불명확해진다.

## Airflow가 해결하는 방식

Airflow는 DAG run, task 상태, task log, Graph View를 통해 파이프라인 실행 상태를 보여준다.
각 task가 분리되어 있으면 실패 지점이 task 단위로 드러나고, 해당 task의 log에서 원인을 확인할 수 있다.

## 프로젝트 적용

### 공통

- [어떤 DAG run이 성공/실패했는지 알아야 함] => Airflow UI의 DAG run 상태로 확인한다.

- [어느 단계에서 실패했는지 알아야 함] => Graph View에서 실패한 task를 확인한다.

- [실패 원인을 확인해야 함] => task log에서 API 오류, timeout, DuckDB lock, 파싱 오류를 확인한다.

- [재실행 범위를 결정해야 함] => 실패한 task와 downstream task를 기준으로 clear/re-run 범위를 정한다.

## 현재 한계

- [Airflow UI 수동 확인에 의존함] => 운영 알림이나 외부 모니터링은 아직 없다.

- [실패를 자동으로 알려주지 않음] => email/slack alert 설정이 필요하다.

- [데이터 품질 이상을 직접 감지하지 않음] => row count, null count, 중복 key 같은 데이터 품질 체크 task가 필요하다.

- [수집 공백을 자동으로 알려주지 않음] => 마지막 성공 수집 시각을 기준으로 gap detection이 필요하다.

## 핵심 문제

로컬 환경에서는 컴퓨터가 꺼져 있으면 Airflow Scheduler도 멈춘다.
예를 들어 12시에 컴퓨터를 끄고 1시에 다시 켜면, 그 1시간 동안 `KIS 현재가` 1분 수집과 `MK RSS` 10분 수집은 실행되지 않는다.
이 문제는 task 재시도가 아니라, 수집되어야 할 시간 구간이 비어 있는지 확인하는 데이터 연속성 문제다.
