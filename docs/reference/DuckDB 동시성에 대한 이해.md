## DuckDB 동시성에 대한 이해

> https://duckdb.org/docs/current/connect/concurrency

---

### 🔹 DuckDB의 동시성 모델

- DuckDB에서 DB 파일을 여는 방식에 따라 동시 접근 가능 범위가 달라짐
  1. 읽기-쓰기 모드 : 하나의 프로세스가 DB를 읽고 쓸 수 있음
     - 하나의 쓰기 프로세스 내부에서 여러 스레드가 쓰기 작업을 할 수 있음(MVCC & 낙관적 동시성 제어)
  2. 읽기 전용 모드 : 여러 프로세스가 DB를 읽을 수 있지만, 쓸 수는 없음
- DuckDB에서 이런 동시성 모델을 사용하는 이유
  - 분석 쿼리를 더 빠르게 실행하려고 쿼리마다 디스크에서 읽고 쓰지 않고 RAM에 캐싱하기 위해
  - 작은 트랜잭션을 많이 실행하는 것이 주된 설계 목표가 아님

---

### 🔹 단일 프로세스에서 DuckDB에 쓰는 규칙

- 쓰기 충돌이 없다면, 여러 스레드의 동시 쓰기 작업은 성공함
- Append 작업은 같은 테이블에 수행되더라도 충돌하지 않음
- 여러 스레드가 서로 다른 테이블을 동시에 업데이트하거나, 같은 테이블의 서로 다른 부분을 동시에 업데이트 가능
- 낙관적 동시성 제어는 두 개의 스레드가 동시에 같은 row를 수정하거나 삭제하려고 할 때 적용됨
  - 이 경우, 나중에 수정을 시도한 두 번째 스레드는 충돌 오류가 발생하면서 실패됨

---

### 🔹 여러 프로세스에서 DuckDB에 쓰는 방법 : DuckLake

- DuckDB의 기본 `.duckdb` 파일에는 여러 프로세스가 동시에 쓸 수 없음
- 동일한 DB에 여러 프로세스가 쓰기 작업을 하려면 DuckLake 형식을 고려할 수 있음
  - 동시 write 문제를 `.duckdb` 단일 파일이 아니라 “데이터 파일 + PostgreSQL catalog” 구조로 풀고, 분석 쿼리 실행은 여전히 DuckDB 엔진이 담당하게 하는 방식
  - 장점 : 동시 쓰기 작업 가능, 분석 쿼리 성능
  - 단점 : 운영 복잡도 증가, DuckLake 성숙도

---

### 🔹 DuckDB는 낙관적 동시성 제어를 사용한다.

- 여러 작업이 충돌하지 않을 것이라고 보고 먼저 실행한 뒤, 실제로 같은 row를 동시에 수정한 경우에만 나중에 충돌 오류를 발생시키는 낙관적 동시성 제어를 사용
- 낙관적 동시성 제어는 읽기 쿼리 처리 속도가 빠름
- 같은 행을 동시에 수정하는 트랜잭션이 발생하면 트랜잭션 충돌 오류 발생
  - 트랜잭션 충돌이 발생하면, 해당 트랜잭션을 다시 실행하면 됨
